SoftN · QuizN 실험실 포털

SoftN Labs 서비스 기획서

QuizN의 실험적 기능·프로토타입(모델)을 공개하고 회원의 의견·별점 피드백을 수집하는 독립 포털. 본 문서는 현재 구현된 화면과 라이브 사이트를 기준으로 기획 의도·정책·구조를 정리한다.

버전 v1.0 (기획) 기준일 2026-06-23 작성 가영 · gy.lee@softn.kr 🌐 gylee209-oss.github.io/quizn-labs

1 문서 개요

목적. 현재 Labs 화면·라이브 사이트를 기준으로 서비스의 기획 의도, 사용자·권한, 정보 구조, 화면별 요구사항, 핵심 정책을 한 문서로 공유한다. 개발 구현 세부(코드)는 별도 문서(labs-md.html)가 담당하며, 본 문서는 왜·무엇을·어떤 규칙으로에 집중한다.

범위. Labs 포털의 진입·인증, 모델 목록·상세, 모델 등록, 권한 관리 4개 핵심 화면과 전역 공통(헤더·푸터·테마), 상태·카테고리·의견 정책. 실제 백엔드 연동은 비범위(정적 시안 단계).

본 기획서는 그간 작성된 PRD·흐름도·와이어프레임·IA(피그마)와 실제 구현 화면을 종합해 역으로 정리한 기준 문서다. 정책 변경 시 본 문서를 먼저 갱신한다.

2 배경 & 문제

QuizN은 다양한 실험 기능·프로토타입을 만들지만, 이를 사용자에게 노출하고 반응을 모으는 일관된 창구가 없었다. 실험 기능은 정식 출시 전 여러 단계(베타·검토 대기·보관)에 흩어져 있어, 상태를 명확히 보여주고 그에 맞게 상호작용을 제어할 장치가 필요했다.

  • 문제 1. 실험 기능의 공개·피드백 수집 채널 부재 → 사용자 반응을 정량·정성으로 모으기 어려움.
  • 문제 2. 기능별 성숙도(상태)가 사용자에게 전달되지 않아 기대치 관리가 안 됨.
  • 문제 3. 사내 자산이므로 접근 통제(로그인 회원·운영자)가 필요.
  • 문제 4. 실험이 늘면서 분류 없는 단순 나열은 탐색이 어려움 → 카테고리 도입 필요.

해결 방향. 상태·카테고리로 구조화한 독립 포털 Labs를 만들어, 로그인 회원이 모델을 탐색하고 의견·별점으로 참여하며, 운영자가 모델을 등록·관리한다.

3 목표 & 성공지표

목표 (Goals)

  • 실험 모델을 상태·카테고리로 구조화해 한 목록에서 탐색·필터·정렬한다.
  • 권한 3단계(비로그인·일반·운영)로 접근·상호작용 범위를 일관되게 제어한다.
  • 회원이 의견·별점으로 참여하고, 비활성(보류·보관) 모델은 읽기 전용으로 보존한다.
  • 운영자가 모델 등록·수정·삭제와 사용자 권한을 관리한다.

성공지표 (KPI · 제안)

구분지표비고
참여모델당 의견 수, 별점 참여자 수핵심 선행지표
탐색상태·카테고리 필터 사용률, 검색 사용률구조화 효과 검증
성숙베타 → 정식 승격 모델 수, 보류·보관 정리율후행지표
⚠ 현재는 정적 시안으로 지표 측정 도구가 미연동이다. 위 KPI는 정식 단계에서 백엔드 이벤트 로깅 연동을 전제로 한 가설·제안이다.

4 사용자 & 권한

권한은 localStorage.labs_auth = guest | member | staff 3단계로 관리한다. 계정 정체성은 quizn_account = {nick, id} 단일 소스를 따르며 기본값은 퀴잰용용 / gy.lee@softn.kr이다.

비로그인 · guest

게이트(잠금) 오버레이만 노출. 콘텐츠 접근 불가. 진입 플로우로만 member 전환.

일반 · member

모델 목록·상세 열람, 의견·별점(내 참여) 참여. 등록·관리 메뉴는 숨김.

운영 · staff

일반 권한 전체 + 새 모델 등록·수정·삭제, 헤더 "관리"(권한 관리).

헤더 Member/Staff 토글로 역할 전환(로그인 상태). 본인 계정은 강등·삭제 불가.

7 화면별 기획

7-1. Labs 목록 labs.html · 허브

목적. 진행 중인 실험 모델을 한눈에 탐색·필터·정렬하고 상세로 진입시키는 중심 화면.

  • 구성: 헤더 · 히어로("Labs 실험실") · 주의 배너 · 컨트롤 바(검색·상태 필터·정렬) · 카테고리 필터(별도 줄) · 모델 카드 그리드(4열) · 페이지네이션(8개 단위).
  • 상태 필터 칩: 전체 · 베타 · 정식 · 보류 · 보관 (카운트 표시).
  • 카테고리 필터: 전체 · 퀴즈앤 확장기능 · 아이디어.
  • 정렬: 최근 업데이트 / 평점 높은순 / 참여 많은순 / 이름순.
  • 카드: 상태 배지 + 카테고리 배지 + 썸네일 + 모델명 + 설명 + 별점(★ 호응도·참여 수). 클릭 → 상세.
  • "진행 중인 모델" 카운트는 비활성(보류·보관)을 제외하고 센다.
  • 운영자에게만 "+ 새 모델 등록" 버튼 노출.

7-2. 모델 상세 labs-detail.html?model={key}

목적. 모델 정보를 전달하고 회원의 의견·별점 참여를 받는다.

  • 상단: 썸네일 + 상태·카테고리 배지 + 모델명 + 모델 키 + 설명 + 액션(외부 데모 열기 / 공유).
  • 통계 3분할: 호응도(평균 별점) · 배포(버전) · 의견(수).
  • 호응도 분포 막대 + 내 참여(별점 클릭 입력 · 수정/삭제).
  • 탭: 의견(별점+의견 목록·페이지네이션) · 공지 · 배포 이력.
  • 의견: 본인 의견만 수정·삭제(확인 모달). 입력은 안전하게 처리(XSS 방지·한글 입력 가드).
  • 연결 방식: 미제공 → 데모 버튼 숨김 / 체험 → "체험하기" / 그 외 → "외부 데모 열기".
  • 운영자가 등록한 모델이면 우상단 수정·삭제 노출.
보류·보관(비활성) 모델은 "이 모델은 OO 상태입니다" 배너 + 의견·별점·외부 데모 비활성(읽기 전용).

7-3. 모델 등록 labs-write.html · STAFF

목적. 운영자가 Labs에 노출할 모델 카드를 등록·수정한다.

  • 입력: 상태(베타·정식·보류·보관) · 카테고리(퀴즈앤 확장기능·아이디어) · 모델명 · 모델 키(영문) · 한 줄 설명 · 연결 방식 · 연결 URL · 썸네일·설명 이미지.
  • 저장: 등록 모델은 목록 상단에 반영. 작성 중 임시저장 지원. ?edit={key} 모드는 "모델 수정"으로 전환하고 기존 평점·의견을 보존.
  • 검증: 모델명·키 필수, 키는 영문 소문자·숫자·_, 연결 방식이 데모/체험이면 URL 필수.
  • 비운영자 접근 시 "등록 권한 없음" 안내(폼 비표시).

7-4. 권한 관리 labs-admin.html · STAFF

목적. 운영자가 사용자 역할을 부여·회수하고 사용자를 추가한다.

  • 사용자 표: 닉네임 · 이메일 · 소속 · 역할 · 역할 변경(일반/운영 토글) · 삭제. 닉네임·이메일 검색.
  • 본인 행은 (나)로 표시하며 강등·삭제 불가.
  • 사용자 추가 모달: 닉네임 · 이메일 · 비밀번호 · 비밀번호 확인 · 역할.
  • 검증: 비밀번호 8자 이상 · 확인 일치. (시안 단계라 비밀번호 값은 저장하지 않음 — 평문 저장 방지.)

8 핵심 정책

8-1. 모델 상태 라이프사이클

상태의미활성참여진행 중 카운트
베타실험 공개·피드백 수집활성가능포함
정식안정 운영활성가능포함
보류검토 대기·일시 중단비활성불가제외
보관종료·신규 대체비활성불가제외
  • 보류·보관은 정책상 동일(비활성·읽기 전용)하며 라벨·색으로만 구분한다. 의미상 보류는 재개 가능, 보관은 종료에 가깝다.
  • 모든 상태 전이는 운영자의 수동 변경(등록·수정 폼)이다 — 시스템 자동 전이가 아니다.
  • 색: 베타=amber, 정식=green, 보류=violet(신규), 보관=gray. 다크/라이트 토큰으로 정의.

8-2. 카테고리

  • 모델은 퀴즈앤 확장기능 또는 아이디어 하나로 분류한다.
  • 목록 상단에 상태 필터와 별도 줄로 카테고리 필터(전체·확장기능·아이디어)를 둔다.
  • 카드·상세에 카테고리 배지 표시. 등록 폼에서 필수 선택. 미지정 모델은 배지 숨김·'전체'에만 노출.
카테고리 추가 시 한 곳(CATEGORY_KO)만 정의하면 카드·필터·등록 폼에 일괄 반영되는 구조다(확장 용이).

8-3. 의견 · 별점 (참여)

  • 용어 통일: 평균 평점 → 호응도, 피드백·댓글 → 의견, 별점 평가 → 참여, 내 평점 → 내 참여. 별(★) 표기는 유지.
  • 회원은 별점(내 참여)과 의견을 남기고, 본인 의견만 수정·삭제(확인 모달).
  • 비활성(보류·보관) 모델은 참여 불가 — "OO 상태의 모델은 참여할 수 없습니다" 안내.

8-4. 운영 (등록 · 권한)

  • 운영자만 모델 등록·수정·삭제, 권한 관리, 사용자 추가.
  • 사용자 추가 시 비밀번호·비밀번호 확인을 받아 8자 이상·일치를 검증한다.
⚠ 정식 단계에서는 비밀번호를 서버에서 해시 저장해야 한다. 현 시안은 검증 UI만 두고 값은 저장하지 않는다.

10 모달 & 상태

13-1. 모델 상태 운영 정책

상태 전이는 모두 운영자의 수동 변경(등록·수정 폼)이며 자동 전이는 없다. 허용 전이와 기록 규칙은 다음과 같다.

전이트리거 / 조건기록
등록 → 베타신규 등록 기본값—
베타 → 정식승격 기준 충족 + 승인승급 사유·승인자
베타·정식 → 보류검토 대기·일시 중단보류 사유·재개 조건 필수
보류 → 베타·정식재개(조건 해소)재개 사유
베타·정식·보류 → 보관종료·신규 대체종료 사유
  • 보류 vs 보관 운영 구분(정책 확정). 보류 = 재개 전제의 일시 중단(사유·재개 조건 기록, 재개 시 이전 상태 복귀). 보관 = 종료 전제(원칙적으로 재개하지 않음, 재오픈은 예외·사유 기록).
  • 승격 기준(베타→정식, 제안값). 예: 호응도 4.0↑ · 참여 N명↑ · 운영 기간 ○주↑ 중 정해진 임계 충족 + 운영 책임자 승인. (수치는 운영팀 합의 후 확정.)
  • 전이 로그. 누가·언제·왜 상태를 바꿨는지 기록(정식 단계 서버). 데모 단계는 기록 생략.
  • 노출. 보류·보관은 목록 상태 필터에 노출하되 "진행 중" 카운트에서는 제외(현행 유지).

13-2. 참여 모더레이션 정책

참여 = 의견(텍스트) · 별점. (투표는 제거.) 로그인 회원(member·staff)만 참여하며 비활성(보류·보관) 모델은 읽기 전용이다.

  • 평점 무결성. 1인 1평점(내 참여), 본인 수정·삭제 가능. 중복·조작 방지는 정식 단계 서버에서 보장(현재 클라이언트 저장이라 미보장).
  • 의견 수정·삭제. 본인 의견만 수정·삭제(현행). 운영자에게 강제 숨김/삭제(모더레이션) 권한 신설 — 스팸·욕설·부적절 콘텐츠 대응.
  • 신고. 회원이 의견을 신고 → 운영자 검토·조치하는 신고 기능 신설(제안).
  • 금칙어·스팸 필터. 등록 시 자동 필터(정식 단계). 위반 시 등록 차단·검토 큐.
  • 보존·삭제 시 처리. 모델 보관 시 기존 의견은 읽기 전용 보존. 모델 삭제 시 누적 의견·평점 처리(완전 삭제 vs 익명 보존)는 운영 정책으로 확정 필요.
투표 제거에 따라 상세 탭 구성은 의견 · 공지 · 배포 이력으로 정리한다.

13-3. 지표 · 측정 규칙

측정은 정식 단계의 백엔드 이벤트 로깅을 전제로 한다(현재 미연동). 지표·산식·수집 이벤트를 명시한다.

지표정의 / 산식수집 이벤트 · 주기
모델당 의견 수기간 내 등록된 의견 합 / 모델의견 등록 · 주
별점 참여자 수 · 참여율고유 참여자 수, 참여자/열람자별점 등록·상세 열람 · 주
평균 호응도유효 별점 평균(1인 1회)별점 등록 · 주
필터·검색 사용률상태·카테고리 필터/검색 사용 세션 비율필터·검색 클릭 · 주
베타→정식 승격 수·율승격 모델 / 베타 모델상태 전이 · 월
보류·보관 정리율보류·보관 전이 모델 / 전체상태 전이 · 월
  • 측정 규칙. 별점은 1인 1회로 중복 제거. "진행 중" 관련 지표는 비활성(보류·보관) 제외. 참여자는 로그인 식별 기준(익명 집계 금지).
  • 리포팅. 주간(참여·탐색)·월간(승격·정리) 단위. 책임=운영팀. 대시보드는 향후 과제.
⚠ 위 임계값·주기는 제안값이다. 운영팀 합의로 확정 후 본 정책을 갱신한다.
